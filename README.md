# claude-skills

## 설치

Claude Code에 이걸 붙여넣으세요.

```
claude-skills를 설치해줘:
mkdir -p ~/.claude/skills/_install && curl -sL https://raw.githubusercontent.com/jyeokchoi/claude-skills/main/skills/_install/SKILL.md -o ~/.claude/skills/_install/SKILL.md
위 명령 실행 후 ~/.claude/skills/_install/SKILL.md 를 읽고 따라해.
```

## 뭐가 들어있나

설치 후 에이전트에게 물어보세요.

## Claude의 조언

이 스킬들은 처음부터 이렇게 설계된 게 아니다. 작업하다가 반복이 보이면 뽑아내고, 안 되면 고치고, 고치다 보니 이렇게 된 거다. 지금도 완성형이 아니다. 그래서 하고 싶은 말은 이거다:

### 1. 100점짜리 워크플로우를 셋업하고 시작하려 하지 마라

가장 흔한 실수다. "완벽한 스킬 시스템을 만들고 나서 작업을 시작해야지." 그러면 영원히 시작을 못 한다.

그냥 만들어라. 3줄짜리 마크다운이어도 된다. "Step 1: 이거 해. Step 2: 저거 해." 이게 스킬이다. 안 되면 고치면 된다. 고치는 것도 에이전트가 한다. 이 레포의 모든 스킬이 정확히 그 과정을 거쳤다 — 처음에는 전부 하드코딩이었고, 쓰면서 일반화했고, 지금도 계속 고치고 있다.

### 2. 스킬은 스킬이 만들게 두어라

스킬을 직접 작성할 필요 없다. 에이전트에게 "방금 내가 한 작업을 스킬로 만들어줘"라고 하면 된다. PR을 3개 올렸으면 그 과정을 스킬로 뽑아달라고 해라. 코드리뷰를 했으면 그 판단 기준을 정리해달라고 해라. 스킬이 스킬을 만들고, 스킬이 스킬을 고친다.

이게 가능한 이유는 스킬이 결국 마크다운이기 때문이다. SKILL.md를 열어보면 에이전트한테 "이 순서로 이렇게 해"라고 쓴 글일 뿐이다. 특별한 프레임워크가 아니다. 에이전트가 읽고 쓸 수 있는 포맷이라는 게 핵심이다.

### 3. 작동 안 해도 상관없다

이 레포의 스킬들도 가끔 이상한 짓을 한다. 엣지 케이스를 다 못 잡고, 에이전트가 멋대로 '개선'하기도 한다. 그래도 쓸만하다. 100번 중 80번 잘 되면 충분하다. 나머지 20번은 그때 고치면 된다.

중요한 건 **안 되는 걸 두려워하지 않는 것**이다. 스킬이 실패하면 에이전트에게 "왜 실패했는지 분석하고 스킬을 고쳐줘"라고 하면 된다. 실패가 곧 개선의 입력이다. 처음부터 모든 케이스를 커버하려고 하면 아무것도 못 만든다.

### 4. 멀티세션은 파일로 풀어라

Claude Code 세션은 끊긴다. 컨텍스트가 날아간다. 이걸 극복하는 가장 확실한 방법은 **파일에 상태를 쓰는 것**이다.

이 레포의 worklog가 그 역할을 한다. 대시보드에 현재 상태를 쓰고, 타임라인에 기록을 남기고, 다음 세션에서 읽어서 이어간다. 마크다운 파일 하나다. "worklog를 업데이트하지 않으면 작업을 완료한 것이 아니다"라는 규칙 하나가 전체를 지탱한다.

### 5. 실전에서 배운 팁 몇 가지

- **하지 말아야 할 것을 적어라** — 에이전트는 과잉 친절하다. "리팩토링하지 마라", "이 파일은 건드리지 마라" 같은 제약이 없으면 알아서 '개선'해버린다
- **설정은 코드에 박지 마라** — 프로젝트별 설정 파일을 두고 스킬이 읽게 하면 된다. 없으면 자동 탐지, 그래도 모르면 물어보고 기억하게 하면 된다
- **반복되는 절차는 `_shared/`로 빼라** — "Shared: `.claude/skills/_shared/create-pr.md` 절차를 따른다"라고 참조만 하면 에이전트가 읽고 따른다
- **판단 기준을 명확히 줘라** — "HIGH 이상 이슈가 없으면 통과"처럼 기준을 주면 에이전트가 멈추지 않고 진행한다

### 요약

일단 만들어라. 에이전트에게 만들게 시켜라. 안 되면 고쳐라. 에이전트에게 고치게 시켜라. 한 달이면 자기만의 워크플로우가 만들어진다.
