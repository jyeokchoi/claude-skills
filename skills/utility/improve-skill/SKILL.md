# 스킬 개선

스킬이 제대로 작동하지 않았거나 개선할 수 있을 때, 사용자와 인터랙티브하게 문제를 파악하고 스킬을 수정한다.

## 경로 규칙

> **`_shared/X`** → `{Base directory}/../_shared/X` (`{Base directory}`는 시스템이 주입하는 "Base directory for this skill" 값)
> **`X` 스킬** → 스킬 시스템이 제공하는 경로. `Glob("**/X/SKILL.md")`로 탐색 가능.

## 트리거

- "스킬이 안 됐어", "스킬 고쳐줘", "스킬 개선해줘"
- "이 스킬이 제대로 작동 안 해", "스킬 수정해줘"
- `/improve-skill`

## 입력

- `skill-name` (선택): 개선 대상 스킬 이름. 미제공 시 대화 맥락에서 자동 감지.

## 절차

### 1. 대상 스킬 식별

다음 우선순위로 대상 스킬을 결정한다:

**1순위 — 인자로 스킬 이름이 주어진 경우:**
- `Glob("**/X/SKILL.md")`로 스킬 파일 탐색

**2순위 — 대화 맥락에서 자동 감지:**
- 현재 대화에서 최근 호출된 스킬이 있으면 (`<system-reminder>` 태그의 스킬 호출 기록, 사용자가 언급한 스킬명, 직전 작업에서 사용된 `/스킬명` 등) 해당 스킬을 대상으로 추정
- 추정된 스킬을 사용자에게 확인: `"'{skill-name}' 스킬을 개선하려는 것이 맞나요?"` (예/아니오)
- "아니오" 시 → 3순위로

**3순위 — 사용자에게 질문 (위 방법으로 특정 불가 시만):**
```
AskUserQuestion:
  question: "어떤 스킬을 개선하고 싶으신가요?"
  header: "대상 스킬"
  description: "스킬 이름을 입력하세요 (예: vwork, pr-review, create-pr)"
```

스킬 이름이 확정되면 `Glob("**/X/SKILL.md")`로 파일 경로를 탐색한다. 미발견 시 사용자에게 알리고 재입력 요청.

### 2. 스킬 현황 파악

대상 스킬의 `SKILL.md`를 `Read`로 전체 읽기.

사용자에게 스킬 요약을 간략히 보여준다:
- 스킬 이름, 설명
- 주요 단계 (##/### 헤딩 기준)
- 허용 도구
- 파일 길이

### 3. 문제 수집 (인터랙티브)

순차적으로 질문하여 개선 맥락을 수집한다. 각 질문은 `AskUserQuestion`으로 한다.

**Q1. 문제 유형**
```
AskUserQuestion:
  question: "어떤 종류의 개선인가요?"
  header: "개선 유형"
  options:
    - label: "버그 수정"
      description: "스킬이 의도대로 작동하지 않음"
    - label: "동작 개선"
      description: "작동은 하지만 더 나은 방식이 있음"
    - label: "기능 추가"
      description: "새로운 기능이나 단계를 추가하고 싶음"
    - label: "구조 개편"
      description: "스킬의 전체 구조를 재설계하고 싶음"
```

**Q2. 구체적 상황**
```
AskUserQuestion:
  question: "구체적으로 어떤 상황이었나요? 에이전트가 어떻게 행동했고, 어떻게 행동했어야 하나요?"
  header: "상세 설명"
  description: "가능한 한 구체적으로 적어주세요. 에이전트의 실제 행동과 기대 행동의 차이를 설명하면 좋습니다."
```

**Q3. 재현 조건 (버그 수정인 경우만)**
```
AskUserQuestion:
  question: "이 문제가 발생하는 조건이 있나요? (항상 발생 / 특정 조건에서만)"
  header: "재현 조건"
  description: "예: 'fork 기반 PR에서만 발생', '워크트리에서 실행할 때만', '특정 인자 조합에서'"
```

**Q4. 추가 맥락 (선택)**
```
AskUserQuestion:
  question: "추가로 공유하고 싶은 맥락이 있나요? (없으면 엔터)"
  header: "추가 맥락"
  description: "관련 에러 메시지, 로그, 또는 참고할 다른 스킬 등"
```

### 4. 원인 분석

수집한 정보를 바탕으로 스킬의 `SKILL.md`를 분석한다:

1. **문제 위치 특정**: 어떤 단계/섹션에서 문제가 발생하는지 식별
2. **근본 원인 추론**: 스킬 정의의 어떤 부분이 문제를 유발하는지 분석
   - 모호한 지시? (에이전트가 잘못 해석할 여지)
   - 누락된 단계? (필요한 절차가 빠짐)
   - 잘못된 순서? (단계 순서가 부적절)
   - 경로/참조 오류? (파일 경로, 스킬 참조가 틀림)
   - 도구 제약? (allowed-tools에 필요한 도구 누락)
   - 조건 분기 부족? (엣지 케이스 미처리)
3. **영향 범위**: 이 스킬을 참조하는 다른 스킬이 있는지 `Grep`으로 확인

분석 결과를 사용자에게 제시한다:
```markdown
## 분석 결과

**문제 위치**: Step N — {단계 이름}
**근본 원인**: {원인 설명}
**영향 범위**: {이 스킬을 참조하는 다른 스킬 목록 또는 "없음"}

### 제안하는 수정사항
1. {수정 1}
2. {수정 2}
...
```

### 5. 수정 방향 확인

```
AskUserQuestion:
  question: "제안된 수정사항에 동의하시나요? 수정하거나 추가할 부분이 있으면 알려주세요."
  header: "수정 방향 확인"
  options:
    - label: "동의"
      description: "제안대로 진행"
    - label: "수정 필요"
      description: "일부 조정이 필요함 (다음에 설명)"
    - label: "다시 분석"
      description: "다른 관점에서 재분석 요청"
```

- **"수정 필요"**: 추가 입력을 받아 수정사항 조정 후 다시 확인
- **"다시 분석"**: Step 4로 돌아가 재분석
- **"동의"**: Step 6으로 진행

### 6. 적용 방식 선택

```
AskUserQuestion:
  question: "어떤 방식으로 적용할까요?"
  header: "적용 방식"
  options:
    - label: "직접 수정"
      description: "SKILL.md를 직접 수정하고 설치까지 완료"
    - label: "프롬프트 제공"
      description: "다른 세션에서 사용할 수 있는 개선 프롬프트를 출력"
```

### 7-A. 직접 수정

1. `Edit`으로 소스 레포의 `SKILL.md` 수정
2. 수정된 전체 파일을 `Read`로 읽어 사용자에게 변경 diff 요약 제시
3. 사용자 확인 후 설치:
   ```bash
   # 설치 경로 결정 (카테고리 디렉토리 제거)
   cp {source_skill_path}/SKILL.md ~/.claude/skills/{skill-name}/SKILL.md
   ```
4. 영향받는 스킬이 있으면 해당 스킬도 수정 필요 여부 안내

### 7-B. 프롬프트 제공

개선 프롬프트를 생성하여 출력한다. 프롬프트는 다음을 포함:

```markdown
## 스킬 개선 프롬프트

**대상**: `{skill-name}` (`{skill-path}`)

### 배경
{문제 상황 요약}

### 현재 동작
{현재 스킬이 어떻게 동작하는지}

### 기대 동작
{사용자가 원하는 동작}

### 수정 지시사항
{구체적인 수정 내용 — 어떤 섹션을, 어떻게 변경해야 하는지}

### 수정 후 검증
- [ ] {검증 항목 1}
- [ ] {검증 항목 2}

### 영향 범위
{이 스킬을 참조하는 다른 스킬 목록 및 연쇄 수정 필요 여부}
```

프롬프트 출력 후 사용자에게 안내:
> "이 프롬프트를 새 세션에서 사용하여 스킬을 개선할 수 있습니다."

## 절대 규칙

1. **사용자 확인 없이 스킬을 수정하지 않는다.** 모든 수정은 Step 5에서 확인, Step 6에서 방식 선택 후에만 진행.
2. **원본은 소스 레포에서 수정한다.** `~/.claude/skills/`를 직접 수정하지 않고 소스 레포 수정 → 복사.
3. **영향 범위를 반드시 확인한다.** 수정 대상 스킬을 참조하는 다른 스킬이 깨지지 않는지 검증.
4. **모호한 지시를 남기지 않는다.** 스킬 수정 시 에이전트가 오해할 여지가 없도록 명확하게 작성.
5. **구조적 변경은 경고한다.** 단계 추가/삭제/재배치 등 구조 변경 시 사용자에게 영향을 명시.
