# Visualize

어떤 개념이든 (UI, 아키텍처, 플로우, 비교, 타임라인, 상태 머신, 계층 구조 등) 텍스트/이미지/코드로 설명하면, 즉시 브라우저에서 열 수 있는 **단일 self-contained HTML 파일**로 시각화한다.

## 트리거

- "그려줘", "시각화해줘", "비교해서 보여줘", "HTML로 만들어줘"
- "목업", "와이어프레임", "다이어그램", "프로토타입"
- "눈으로 보고 싶어", "정리해서 보여줘", "한눈에 보게 해줘"
- `/visualize`

## 입력 (하나 이상, 조합 가능)

1. **텍스트 설명**: 자연어로 된 아이디어, 요구사항, 비교 항목
2. **이미지**: 기존 UI 스크린샷, 손그림, 화이트보드 사진
3. **코드/데이터**: JSON, 타입 정의, API 스펙 등 구조화된 정보
4. **대화 맥락**: 직전 대화에서 논의한 내용을 시각화

## 출력 규격

- **단일 self-contained HTML** (외부 의존성 절대 없음 — CDN, 외부 이미지, 웹폰트 금지)
- CSS는 `<style>`, JS는 `<script>` 내 인라인
- 경로: `~/visualize/{descriptive-name}.html`
- 파일명은 내용 기반으로 의미 있게 작명 (예: `auth-flow-comparison.html`, `volume-ui-mockup.html`)
- 아이콘이 필요하면 간단한 SVG path로 직접 그림

## 시각화 유형 (자동 선택)

입력을 분석하여 가장 적합한 유형을 **자동 선택**한다. 사용자에게 "어떤 형식으로 할까요?" 묻지 않고 바로 그린다. 판단이 불확실할 때만 확인.

| 유형 | 감지 신호 | 렌더링 전략 |
|------|-----------|-------------|
| **비교형** | "A안 vs B안", 복수 옵션 | 카드 나란히 배치, 차이점에 NEW/CHANGED 인디케이터 |
| **플로우형** | "흐름", "단계", "프로세스", 순서 | 노드 + 화살표, 스텝 번호, 분기점은 다이아몬드 |
| **계층/관계형** | "구조", "의존성", "계층" | 중첩 박스, 연결선, 들여쓰기 트리 |
| **매트릭스/표형** | "비교표", "장단점", "트레이드오프" | 스타일링 테이블, 셀 내 아이콘/색상 |
| **UI 목업형** | "UI 그려줘", "화면", "패널/모달" | 실제 UI처럼 보이는 인터랙티브 컴포넌트 |
| **타임라인형** | "일정", "마일스톤", "히스토리" | 수평/수직 타임라인, 노드 + 날짜 + 설명 |
| **자유 형식** | 위 어디에도 맞지 않을 때 | 정보 구조에 맞는 창의적 조합 |

복합 유형도 가능 (예: 비교형 + UI 목업형).

## 디자인 원칙

1. **내용이 형식을 결정한다** — 정보 구조를 먼저 파악하고, 그에 맞는 레이아웃을 선택
2. **차이를 강조한다** — 같은 부분은 조용하게, 다른 부분은 색상/뱃지/보더로 시선 유도
3. **현실적 더미 데이터** — placeholder, lorem ipsum, "항목 1" 금지. 맥락에 맞는 구체적 예시 사용 (예: `summer_vibes_bgm.mp3`, `김지연`)
4. **정보 밀도 ≥ 장식** — 불필요한 장식 없이 정보 전달에 집중. 여백은 구조를 드러내기 위해 사용
5. **한 눈에 파악** — 스크롤 없이 핵심이 보이면 최선. 불가피하면 중요한 정보를 상단에 배치

## 스타일 가이드라인

맥락에 맞게 조정하되, 일관성을 위한 기본값:

```css
font-family: -apple-system, BlinkMacSystemFont, 'Segoe UI', sans-serif;
/* 페이지 배경: #f0f0f0 / 카드 배경: #fff */
/* 색상: 의미 기반 할당 — 비교 안마다 고유 색, 상태별 색 (성공=#22c55e, 경고=#f59e0b, 에러=#ef4444) */
/* 카드: border-radius: 8px, box-shadow: 0 2px 12px rgba(0,0,0,0.12) */
/* 반응형: flex-wrap: wrap */
```

**이미지 입력으로 기존 UI가 주어지면** 해당 UI의 스타일을 우선 재현하고, 이 가이드라인은 보조적으로만 사용한다.

## 실행 프로세스

### 1. 입력 분석

- 이미지 있음 → 기존 시각 구조 파악 (레이아웃, 색상, 컴포넌트, 라벨)
- 텍스트 있음 → 핵심 개념, 관계, 비교 축 추출
- 코드/데이터 있음 → 구조와 관계 파싱
- 대화 맥락 → 직전 논의에서 시각화 대상 식별

### 2. 시각화 유형 결정

- 위 7가지 유형 중 가장 적합한 것 선택 (복합 가능)
- 유형을 사용자에게 명시적으로 언급 ("비교형으로 그리겠습니다")
- 사용자가 원하는 형식을 지정했으면 그것을 따름

### 3. HTML 생성

```bash
mkdir -p ~/visualize
```

- `Write` 도구로 `~/visualize/{descriptive-name}.html` 생성
- 현실적 더미 데이터 채움
- 차이점/핵심에 시각적 강조 적용

### 4. 브라우저에서 열기

```bash
open ~/visualize/{descriptive-name}.html
```

### 5. 수정 반복 (사용자 피드백 시)

- `Edit` 도구로 해당 부분만 수정 — 전체 재작성 금지, 점진적 개선
- 매 수정 후 변경 사항 bullet 요약
- 수정 후 다시 `open`으로 열기

## 절대 규칙

1. **Self-contained**: 외부 리소스 로드 절대 없음. 모든 것이 파일 안에 있어야 함.
2. **즉시 렌더링**: `open {파일경로}`로 바로 열 수 있어야 함.
3. **자동 판별**: 사용자에게 형식을 묻지 않고, 입력에서 최적 유형을 판단하여 바로 그린다.
4. **점진적 수정**: 피드백마다 `Edit`으로 해당 부분만 수정. 전체를 다시 쓰지 않는다.
5. **설명 < 시각화**: 텍스트로 장황하게 설명하지 않는다. HTML이 스스로 말하게 한다.
6. **한국어 기본**: 모든 라벨, 설명, 주석은 한국어. 코드 식별자와 파일명만 영문.
7. **맥락 충실**: 대화 중 논의된 도메인 용어, 구조, 네이밍을 그대로 사용한다.
