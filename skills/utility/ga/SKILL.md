# GA (Analytics Tracking)

## 경로 규칙

> **`_shared/X`** → `{Base directory}/../_shared/X` (`{Base directory}`는 시스템이 주입하는 "Base directory for this skill" 값)
> **`X` 스킬** → 스킬 시스템이 제공하는 경로. `Glob("**/X/SKILL.md")`로 탐색 가능.

## 목적

워크로그와 현재 브랜치의 코드 변경사항을 분석하여, 서비스/사용자 관점에서 트래킹할 가치가 있는 액션을 식별한다. 사용자 승인 후 기존 코드의 동작 변경 없이 analytics 코드를 삽입하고, 결과를 워크로그에 기록한다.

## 핵심 원칙

1. **기존 동작 무변경**: 트래킹 코드 추가로 인해 기존 로직이 변경되어서는 안 된다
2. **아키텍처 비오염**: 기존 코드 구조에 새로운 추상화/레이어/의존성을 도입하지 않는다. 코드베이스에 이미 존재하는 analytics 인프라와 패턴을 그대로 따른다
3. **사용자 승인 필수**: 이벤트 목록을 먼저 제안하고, 사용자가 승인/수정한 후에만 코드에 반영한다

## 단계

### 1. 워크로그 및 변경사항 수집

`_shared/resolve-worklog-target.md`를 로드하고 해당 절차를 따른다 (`required_files`: 없음).

워크로그에서 추출할 정보:
- **Goal**: 작업의 목적 (어떤 기능/개선이 이루어졌는지)
- **Decisions**: 핵심 결정사항 (트래킹 대상 판단에 참고)
- **Timeline**: 최근 작업 내역

브랜치 diff 수집:

```bash
# base branch 결정
BASE_REF=$(git merge-base HEAD origin/main 2>/dev/null || git merge-base HEAD main 2>/dev/null)

# 변경된 파일 목록
CHANGED_FILES=$(git diff --name-only $BASE_REF..HEAD)

# 변경 통계
DIFF_STAT=$(git diff --stat $BASE_REF..HEAD)
```

### 2. 기존 Analytics 인프라 탐색

코드베이스에서 기존 트래킹 패턴을 탐색한다. **이 단계의 결과가 Step 4의 구현 방식을 결정한다.**

탐색 대상 (병렬 Grep):

| 패턴 | 목적 |
|------|------|
| `analytics`, `tracking`, `telemetry` | 전용 모듈/서비스 |
| `track(`, `logEvent(`, `sendEvent(`, `emit(` | 이벤트 전송 함수 |
| `gtag(`, `ga(`, `mixpanel`, `amplitude`, `segment` | 외부 서비스 SDK |
| `useAnalytics`, `useTracking`, `AnalyticsProvider` | React hooks/context |
| `@track`, `@analytics`, `@log` | 데코레이터 |
| `event_name`, `eventName`, `action_type` | 이벤트 정의 |

탐색 결과를 정리:

```markdown
## 기존 Analytics 인프라 요약

- **전송 방식**: {함수명/모듈명} (예: `analytics.track()`, `gtag('event', ...)`)
- **이벤트 정의 위치**: {파일 경로} (예: `src/analytics/events.ts`)
- **호출 패턴**: {코드 예시}
- **이벤트 네이밍 컨벤션**: {snake_case/camelCase/기타}
- **파라미터 구조**: {공통 파라미터 패턴}
```

**인프라 미발견 시**: 사용자에게 알리고 어떤 analytics 도구를 사용할지 확인한다. 새로운 인프라 구축은 이 스킬의 범위 밖이다.

### 3. 트래킹 이벤트 식별 및 제안

변경사항과 워크로그 Goal을 기반으로, **서비스/사용자 관점**에서 트래킹할 가치가 있는 액션을 식별한다.

**무거운 분석은 위임한다** — `_shared/delegation-policy.md` 참조. 변경 파일이 3개 이상이면 `explore` 에이전트에게 위임하여 사용자 액션을 식별한다.

식별 기준:

| 트래킹 가치 높음 | 트래킹 가치 낮음 |
|------------------|------------------|
| 사용자가 명시적으로 수행하는 액션 (클릭, 제출, 토글) | 내부 상태 변경, 리렌더링 |
| 비즈니스 KPI에 직결되는 이벤트 (구매, 가입, 전환) | 기술적 에러 로깅 (별도 모니터링 영역) |
| 기능 사용 빈도를 측정할 수 있는 이벤트 | 매 프레임/매 스크롤 발생하는 이벤트 |
| A/B 테스트 분기점 | 이미 트래킹되고 있는 이벤트 |

사용자에게 제안할 형식:

```markdown
## 트래킹 이벤트 제안

### 추가 예정 ({N}건)

| # | 이벤트명 | 트리거 | 파라미터 | 삽입 위치 | 근거 |
|---|----------|--------|----------|-----------|------|
| 1 | `{event_name}` | {사용자 액션} | `{param: type}` | `{file:line}` | {왜 트래킹하는지} |
| 2 | ... | ... | ... | ... | ... |

### 제외 사유 (참고)

| 변경사항 | 제외 이유 |
|----------|-----------|
| {변경 내용} | {트래킹 불필요 사유} |
```

### 3-A. 사용자 승인

```
AskUserQuestion:
  question: "위 트래킹 이벤트 목록을 검토해주세요. 추가/제거/수정할 항목이 있으면 알려주세요."
  options:
    - label: "승인"
      description: "제안된 목록대로 진행합니다"
    - label: "수정 필요"
      description: "목록을 수정한 뒤 다시 검토합니다"
    - label: "취소"
      description: "트래킹 추가를 중단합니다"
```

- **승인**: Step 4로 진행
- **수정 필요**: 사용자 피드백 반영 후 목록 재출력 → 재승인
- **취소**: 스킬 종료

### 4. Analytics 코드 삽입

승인된 이벤트 목록을 기반으로 코드에 트래킹을 추가한다.

**구현 규칙:**

1. **Step 2에서 발견한 기존 패턴만 사용한다**
   - 기존 import 경로, 함수 시그니처, 네이밍 컨벤션을 그대로 따른다
   - 새로운 유틸리티 함수, 래퍼, 추상화를 만들지 않는다

2. **삽입 위치는 사용자 액션 핸들러 내부이다**
   - 이벤트 핸들러 (`onClick`, `onSubmit`, `onChange` 등) 내부에 1줄 추가
   - 또는 해당 액션을 처리하는 함수의 시작/끝에 추가
   - 조건문이나 제어 흐름을 변경하지 않는다

3. **실패에 안전해야 한다**
   - 트래킹 코드가 throw하면 안 된다 (기존 패턴이 try-catch를 포함하면 동일하게)
   - 비동기 트래킹은 await하지 않는다 (fire-and-forget)

4. **파일별 수정은 최소한으로**
   - 이벤트 전송 1줄 + (필요시) import 1줄만 추가
   - 기존 코드의 인덴테이션, 공백, 포맷팅을 유지한다

**구현 방법:**

변경 파일이 3개 이상이면 `executor` 에이전트에게 위임한다. 프롬프트에 다음을 포함:
- 승인된 이벤트 목록 (이벤트명, 파라미터, 삽입 위치)
- Step 2에서 발견한 기존 analytics 패턴 (import 경로, 함수 호출 형식)
- "기존 동작 무변경, 아키텍처 비오염" 원칙

### 5. 변경 검증

삽입 완료 후 검증:

```bash
# 빌드 확인 (프로젝트에 빌드 명령이 있는 경우)
# 린트 확인 (프로젝트에 린트 명령이 있는 경우)
```

검증 항목:
- [ ] 트래킹 코드 추가로 인한 빌드 에러 없음
- [ ] 기존 테스트 통과 (트래킹 코드가 동작을 변경하지 않았음을 증명)
- [ ] 각 이벤트의 삽입 위치가 승인 목록과 일치

### 6. 워크로그 업데이트

`_shared/update-worklog.md`를 로드하고 해당 절차를 따른다:

- `timeline_entry`: 트래킹 이벤트 추가 결과

타임라인에 포함할 내용:

```markdown
**Analytics 트래킹 추가**

- **추가된 이벤트 ({N}건)**:
  | 이벤트명 | 파일 | 트리거 |
  |----------|------|--------|
  | `{event_name}` | `{file}` | {사용자 액션} |

- **사용한 인프라**: {analytics 모듈/함수명}
- **검증**: 빌드 {PASS/FAIL}, 테스트 {PASS/FAIL}
```

### 7. 요약 출력

```
Analytics Tracking 완료:
  - 추가된 이벤트: {N}건
  - 수정된 파일: {files}
  - 사용한 인프라: {analytics 패턴}
  - 빌드: {PASS/FAIL}
  - 워크로그: 업데이트 완료
```

## 에러 처리

| 상황 | 처리 |
|------|------|
| 워크로그 미발견 | 워크로그 없이 브랜치 diff만으로 진행 가능 (사용자 확인) |
| Analytics 인프라 미발견 | 사용자에게 알리고 도구 선택 요청. 인프라 구축은 범위 밖 |
| 빌드 실패 | 삽입한 트래킹 코드를 revert하고 사용자에게 보고 |
| 트래킹 대상 없음 | "트래킹할 사용자 액션이 식별되지 않았습니다" 출력 후 종료 |

## 절대 규칙

- **기존 동작을 변경하지 않는다** — 조건문, 제어 흐름, 반환값을 수정하지 않는다
- **새로운 아키텍처를 도입하지 않는다** — 기존 analytics 패턴만 사용한다
- **사용자 승인 없이 코드를 수정하지 않는다** — Step 3-A 승인 필수
- **무거운 작업은 위임한다** — `_shared/delegation-policy.md` 참조
